Section 02

Programming Languages and Their Roles

Four languages are used in this project. Each one is assigned to the tasks it does best. They never overlap in responsibility, which keeps the code clean and easy to debug.

Mathematics
MATLAB
Primary language for all maths, modelling, and simulation
MATLAB is used for everything that involves mathematics, from writing the dynamic equations to running the Simulink simulations to post-processing flight data. It is the only language with the toolboxes needed for symbolic computation, Lyapunov verification, control system design, and Simulink modelling all in one environment. Every formula derived on paper gets validated first in MATLAB before it is sent to hardware.
Phases 1-4 Phase 6 analysis Report figures
Hardware Bridge
Python 3.10
Control loop on the Raspberry Pi companion computer
Python runs the actual SMC algorithm in real time on the Raspberry Pi that sits on the drone. It talks to the Pixhawk using ROS and MAVROS, reads the attitude measurements at 250 Hz, runs the SMC math, and sends motor commands back. Python is chosen here because it has excellent ROS bindings, NumPy for fast matrix math, and it can be edited and restarted during testing far faster than compiled code. The SMC algorithm itself is a direct translation of the MATLAB maths into Python.
Phase 5 live control Log conversion Hardware tests
Middleware
ROS (Robot Operating System)
Message passing framework between all software nodes
ROS is not a programming language itself but a framework that lets separate programs talk to each other by sending typed messages over named topics. On this drone, the MAVROS node (which talks to Pixhawk) and the SMC node (which runs our controller) are separate programs. ROS manages the communication between them so that the SMC node can subscribe to attitude data and publish control commands without needing to know anything about MAVLink protocol details. This separation makes testing and debugging much easier.
Phase 5 integration Node communication Topic publish/subscribe
Firmware
PX4 / C++ (pre-built firmware)
Flight controller firmware — we configure it, not rewrite it
The Pixhawk runs PX4 Autopilot firmware written in C++. We do not modify the PX4 source code. Instead we configure it through QGroundControl and then command it from our Python SMC node over OFFBOARD mode via MAVLink. PX4 handles all the safety-critical low-level tasks: ESC signal generation, failsafe monitoring, GPS fusion, and hardware-level sensor reading. This division of responsibility is essential: PX4 handles the hardware and we handle the control algorithm.
Phase 5 hardware ESC control Sensor fusion
Section 03

MATLAB Toolboxes — What Each One Does

A MATLAB toolbox is a collection of pre-built functions for a specific domain. Without the right toolboxes you cannot run certain commands. Here is every toolbox this project uses, what it contains, and which project phase needs it.

Required — Phase 2
Control System Toolbox
MATLAB add-on
This toolbox provides functions for designing, analysing, and simulating classical and modern linear control systems. For this project it is used to linearise the nonlinear quadcopter model around the hover operating point, compute the transfer function for each decoupled axis, analyse stability margins using Bode and Nyquist plots, and calculate initial PID gains. The key function linearize() extracts state-space matrices A and B from the Simulink model automatically, saving the manual algebra of computing Jacobians by hand.
linearize() pidtune() bode() stepinfo() margin() tf() ss()
Required — Phase 3
Symbolic Math Toolbox
MATLAB add-on
This toolbox lets MATLAB manipulate mathematical expressions symbolically rather than numerically. Think of it as a built-in computer algebra system similar to Mathematica. For this project it is used in lyapunov_verify.m to check that the time derivative of the Lyapunov function V equals half s squared is truly negative when the control law is substituted in. The toolbox handles the chain rule differentiations and algebraic simplifications automatically, so any error in the hand derivation is caught before the Simulink simulation is ever built.
syms diff() simplify() subs() assume() solve()
Required — Phase 4
Simulink (base product)
MATLAB graphical simulation environment
Simulink is not a toolbox but MATLAB's graphical block-diagram simulation environment. It is where the entire closed-loop control system is built and tested. The nonlinear plant, PID controller, SMC controller, reference signal generator, and disturbance injector are all separate blocks connected by signal lines. Simulink runs the simulation using a numerical solver under the hood, but the engineer sees the system as a block diagram rather than written code, which makes the architecture much easier to understand and explain in a report.
sim() MATLAB Fn Block Step Block To Workspace ode4 solver
Required — Phase 4
Simulink Control Design Toolbox
MATLAB add-on
This toolbox extends Simulink with tools for tuning and analysing controllers directly within a Simulink model. The most useful feature for this project is the ability to compute the operating point (the hover condition) from within a running Simulink model using findop(), and then linearise the entire model at that operating point using linearize(). This gives the linear A and B matrices directly from the graphical model without requiring any manual derivation of the Jacobian matrices, saving significant time and reducing the chance of errors.
findop() linearize() PID Tuner app Operating point spec
Required — Phase 6
Statistics and Machine Learning Toolbox
MATLAB add-on
This toolbox provides statistical analysis functions needed to properly validate the experimental results. For this project, the most critical function is ttest() which performs the paired t-test that determines whether the performance difference between SMC and PID is statistically significant at the 5 percent level. Without statistical testing, reporting that SMC is better than PID carries no academic weight because the difference might be due to random variation. This toolbox makes the experimental analysis rigorous and defensible to examiners.
ttest() mean() std() normplot() boxplot()
Optional — Phase 1
Aerospace Toolbox
MATLAB add-on
This toolbox provides aerospace-specific functions including standard atmosphere models, coordinate transformations, and quaternion rotation utilities. For this project it is helpful for verifying the rotation matrix computation in rotation_matrix.m using the toolbox's built-in angle2dcm() function as a cross-check. It also provides the angle2quat() and quat2angle() functions which are useful if the project later considers quaternion-based attitude representation as an alternative to Euler angles.
angle2dcm() angle2quat() dcm2angle() Cross-check rotation
Section 04

Essential MATLAB Commands Explained

These are the specific MATLAB commands used most frequently in this project. Each one is explained so you know exactly what it does and why it is needed, not just how to type it.

Phase 1 — Model Validation Commands
MATLAB
»options = odeset('RelTol', 1e-8, 'AbsTol', 1e-10);
Sets the ODE solver tolerance. RelTol controls how accurate each integration step must be relative to the solution magnitude. AbsTol controls accuracy when the solution is near zero. Tight tolerances are essential for nonlinear systems because the default (1e-3) can miss small but physically important coupling effects.
»[t, x] = ode45(@(t,x) dynamics(t, x, u, params), t_span, x0, options);
The main simulation call. ode45 uses a 4th/5th order Runge-Kutta method to integrate the ODE forward in time. The @(t,x) part is an anonymous function handle that passes the current time and state to our dynamics.m file at every solver step. t_span is [0, 5] for 5 seconds. x0 is the initial state (zeros for starting at rest).
»phi_deg = rad2deg(x_out(:, 1));
Converts the roll angle column from radians (what the ODE uses internally) to degrees (what humans read). The colon operator x(:,1) selects the entire first column — all timestep values of the first state variable (phi).
Phase 2 — PID Design Commands
MATLAB + Control System Toolbox
»op = operpoint('pid_simulink');
Finds the hover operating point inside the Simulink model called pid_simulink.slx. An operating point is the set of state values and inputs where all derivatives are zero (equilibrium). For this drone that is the hover condition where all angles and rates are zero.
»sys = linearize('pid_simulink', op);
Automatically computes the Jacobian matrices A, B, C, D at the operating point by numerically perturbing each state and input a tiny amount and measuring how the output changes. This is much less error prone than computing the Jacobians by hand and gives you a standard state-space object you can use with all Control System Toolbox functions.
»info = stepinfo(y, t, 'SettlingTimeThreshold', 0.02);
Automatically extracts the step response metrics from a time-series output y. The SettlingTimeThreshold of 0.02 means settling is defined as staying within 2 percent of the final value. Returns a struct with fields: RiseTime, SettlingTime, Overshoot, Peak, and SteadyStateValue. This is used in compare_results.m to build the performance table without manually reading values off plots.
»[Gm, Pm, Wgm, Wpm] = margin(sys_open_loop);
Computes the gain margin Gm and phase margin Pm of the open-loop transfer function. Gain margin should be above 6 dB and phase margin above 45 degrees for a robustly stable linear system. If these margins are too small, disturbances or model uncertainty can cause the PID closed loop to go unstable.
Phase 3 — Symbolic Lyapunov Verification
MATLAB + Symbolic Math Toolbox
»syms s e edot lambda K D eta real positive
Declares symbolic variables with the assumption that they are all real and positive. This lets MATLAB simplify expressions correctly — for example it knows that K minus D is positive when K is greater than D, which is exactly the assumption used in the stability proof.
»V = (1/2) * s^2;
»Vdot = simplify(diff(V, s) * sdot_closed_loop);
Computes the time derivative of the Lyapunov function using the chain rule: V_dot equals dV/ds times ds/dt. The simplify() function reduces the result to its simplest algebraic form. We then check that this expression is always negative by confirming it equals something like -eta times abs(s) or less.
»is_negative = isAlways(Vdot <= -eta * abs(s));
Checks symbolically whether the condition V_dot is less than or equal to negative eta times abs(s) is always true given the assumptions. If this returns true, the Lyapunov stability proof is verified algebraically and the SMC design is theoretically correct before a single simulation is run.
Phase 4 — Batch Simulation and Results
MATLAB + Simulink
»simOut = sim('full_model', 'StopTime', '10');
Runs the Simulink model from a MATLAB script rather than pressing the play button manually. This is how run_all_tests.m executes all five test cases back-to-back automatically. The results are captured in simOut and can be post-processed immediately in the same script.
»rmse = sqrt(mean((phi_ref - phi_actual).^2));
Computes the Root Mean Square Error between the desired and actual roll angle over the test window. The .^ operator applies squaring element-wise to the whole vector. RMSE is the primary steady-state tracking error metric used in the simulation-to-hardware comparison table.
»exportgraphics(gcf, 'figures/step_response.pdf', 'Resolution', 600);
Exports the current figure (gcf means get current figure) as a high-resolution PDF at 600 DPI. PDF is used rather than PNG because PDF is vector-based, meaning the figure looks perfectly sharp at any zoom level in the report. This is the correct format for academic publication-quality figures.
Phase 6 — Experimental Analysis Commands
MATLAB + Statistics Toolbox
»smc_data = readtable('csv_data/smc_step_01_vehicle_attitude.csv');
Reads the flight log CSV file into a MATLAB table where each column is a variable (timestamp, roll_angle, pitch_angle, etc.). Tables preserve column names which makes the code self-documenting compared to raw matrix indexing.
»[h, p, ci] = ttest(smc_metric, pid_metric);
Performs a two-sided paired t-test comparing the SMC metric vector to the PID metric vector across the repeated trials. h is 1 if the null hypothesis (no difference) is rejected at the 5 percent level. p is the p-value. ci is the 95 percent confidence interval for the difference. If h equals 1 and p is less than 0.05, the performance difference is statistically significant.
»writetable(results_table, 'results_summary.csv');
Writes the final results comparison table to a CSV file. This CSV is the authoritative record of all experimental results and is imported directly into the report tables to ensure no transcription errors occur when copying numbers from MATLAB plots into Word.
Section 05

Python Libraries — What Each One Does

Python on the Raspberry Pi handles the real-time control loop during flight. Each library has a single job, and together they form a lightweight but complete control system that can run comfortably at 250 Hz on a Raspberry Pi 4.

Core — smc_algorithm.py
NumPy
Numerical array and matrix mathematics
NumPy provides the array and matrix operations that the SMC algorithm needs to run efficiently in Python. Without NumPy, every matrix multiplication and element-wise operation in the control law would have to be written as slow Python for-loops. With NumPy, operations like computing the sliding surface vector, applying the saturation function, and multiplying the inertia matrix all run in compiled C code behind the scenes, making the 250 Hz control loop fast enough to meet real-time requirements on the Raspberry Pi.
np.array() np.dot() np.clip() np.sign() Matrix ops
Core — mavlink_bridge.py
MAVROS (via rospy)
MAVLink protocol ROS interface
MAVROS is a ROS package that provides a bridge between the ROS topic system and the MAVLink protocol used by PX4 on the Pixhawk. Rather than implementing MAVLink packet parsing ourselves (which would be a significant engineering task), we simply subscribe to MAVROS topics like /mavros/imu/data for attitude measurements and publish to /mavros/actuator_control to send motor commands. MAVROS handles all the protocol translation transparently. It also provides service calls for arming the drone and switching flight modes.
rospy.Subscriber() rospy.Publisher() Imu message type ActuatorControl msg ServiceProxy arm
Log conversion
pyulog
PX4 ULog file reader and converter
PX4 saves flight logs in its own binary ULog format which cannot be read directly by MATLAB or spreadsheet tools. The pyulog library reads these binary files and extracts the logged topics as Python dictionaries. The convert_logs.py script uses pyulog to batch-convert every ULog file in the flight_logs folder into CSV files that MATLAB can then load with readtable(). This conversion step is the bridge between the hardware data and the MATLAB analysis pipeline.
ULog() to_csv() message_formats
Configuration
PyYAML
YAML file reader for parameter loading
The config.yaml file stores all the SMC tuning parameters (lambda values, switching gains, boundary layer widths) as a simple text file. PyYAML reads this file at startup and passes the parameters to the SMC algorithm. This design means that when the team wants to try different gain values between flight tests, they only need to edit a text file and restart the ROS node — there is no need to touch the Python code at all. This separation of configuration from code is a fundamental software engineering best practice.
yaml.safe_load() config.yaml parsing
Timing
Python time module
Built-in Python standard library
The time module provides the tools needed to enforce the 250 Hz control loop rate. The smc_node.py uses time.perf_counter() to measure exactly how long each control cycle takes and time.sleep() to wait out the remainder of the 4-millisecond window. If a loop iteration takes longer than 4 milliseconds due to system load, the code logs a warning but does not crash. Monitoring these overruns during testing reveals whether the Raspberry Pi has enough processing headroom for the chosen control rate.
time.perf_counter() time.sleep() Loop timing
Diagnostics
Python logging module
Built-in Python standard library
The logging module writes timestamped diagnostic messages to a file during flight. Every time the SMC computes a control output, the sliding surface values, torque commands, and loop timing are written to a rotating log file. This log is invaluable for post-flight debugging: if the drone behaved unexpectedly during a test, you can replay exactly what the SMC was computing at every moment to understand why. The log level (DEBUG, INFO, WARNING) can be set in config.yaml without code changes.
logging.getLogger() RotatingFileHandler Flight diagnostics

Python Setup — Install Commands Explained

These commands are run once on the Raspberry Pi to set up the Python environment. Every command is explained so you understand what it does, not just that it works.

Step 1 — System Update (run first, always)
Raspberry Pi Terminal
$sudo apt update && sudo apt upgrade -y
apt update refreshes the package list from all configured software repositories. apt upgrade installs the latest versions of all currently installed packages. The -y flag auto-confirms all prompts. Running this before installing anything else ensures that all dependency version tables are current and prevents version conflicts later.
Step 2 — Python and Pip
Raspberry Pi Terminal
$sudo apt install python3.10 python3-pip python3-venv -y
Installs Python 3.10 (the version ROS Noetic works best with), pip (the Python package manager), and venv (virtual environment tool). A virtual environment isolates this project's packages from the system Python so that installing pyulog does not break anything else on the Pi.
$python3 -m venv ~/smc_env
$source ~/smc_env/bin/activate
Creates a virtual environment called smc_env in the home directory, then activates it. After activation, any pip install command installs packages only into this isolated environment. The terminal prompt changes to show (smc_env) to confirm it is active. This must be done every time a new terminal session is opened.
Step 3 — Install All Python Packages
Raspberry Pi Terminal (venv active)
$pip install numpy==1.24.3
Installs NumPy version 1.24.3 specifically. Pinning the version number ensures the project behaves the same on every machine and prevents breakage if NumPy releases an update that changes some behaviour. The exact version number comes from the requirements.txt file.
$pip install pyulog==0.9.1
Installs the pyulog flight log converter. Version 0.9.1 is compatible with PX4 firmware v1.14 log format. Using a newer version of pyulog with an older log format can sometimes produce silent parsing errors where some topics are simply absent from the output CSV without any error message.
$pip install pyyaml
Installs PyYAML for reading the config.yaml parameter file. YAML is chosen over JSON for the config file because it supports comments (lines starting with #), which makes it possible to document what each parameter does directly inside the configuration file. JSON does not support comments.
$pip install -r requirements.txt
Installs all packages listed in requirements.txt at once. This is the shortcut command used when setting up on a new machine. The requirements.txt file lists every package with its exact version and is kept up to date every time a new dependency is added using pip freeze > requirements.txt.
Section 07

ROS Packages — What Each One Provides

ROS Noetic is the version of ROS that runs on Ubuntu 20.04, which is what the Raspberry Pi runs. Each ROS package below provides either core infrastructure or a specific capability our drone needs.

Core ROS
ros-noetic-desktop-full
The complete ROS Noetic installation
This meta-package installs the entire ROS Noetic base system including the communication middleware (topics, services, actions), the roscore master process that coordinates all nodes, the build system (catkin), rviz for 3D visualisation, and dozens of standard message types. Installing this one package gives you everything needed to write and run ROS nodes. It is approximately 1.5 GB and takes around 15 minutes to install on a Raspberry Pi 4.
roscore rosrun roslaunch rostopic catkin_make
Hardware Bridge
ros-noetic-mavros
MAVLink to ROS bridge package
MAVROS is the most critical ROS package for this project. It runs as a ROS node that maintains a persistent MAVLink connection to the Pixhawk over a USB or UART serial link. It continuously reads all PX4 telemetry streams (attitude, IMU, GPS, battery, flight mode) and publishes them as ROS topics. Simultaneously, it subscribes to ROS topics from our SMC node and forwards the control commands to PX4 as the appropriate MAVLink messages. Without MAVROS we would need to implement the entire MAVLink protocol from scratch.
/mavros/imu/data /mavros/state /mavros/actuator_control set_mode service arming service
Hardware Bridge
ros-noetic-mavros-extras
Extended MAVROS message types and plugins
This companion package to MAVROS provides additional message types and protocol plugins including support for vision-based position estimation, optical flow sensors, and extended camera control. For this project the most useful component is the GeographicLib datasets that MAVROS needs for coordinate frame transforms. Without installing this package, MAVROS sometimes fails to start because it cannot find the required geographic conversion data, producing a cryptic error message about missing datasets.
GeographicLib data Extended msg types
Our Package
smc_controller_pkg
Custom ROS package (written by us)
This is the ROS package we create ourselves. It contains our SMC node, the SMC algorithm module, the motor mixing module, and the MAVLink bridge wrapper. Creating it as a proper ROS package rather than a standalone Python script means it integrates cleanly with the ROS build system, can be launched with a single roslaunch command, and has proper dependency declarations so that anyone can build it on a fresh machine by just running catkin_make. The package follows standard ROS conventions for folder layout and naming.
smc_node.py smc_flight.launch config.yaml CMakeLists.txt
Section 08

ROS Setup and Flight Commands Explained

Step 1 — Install ROS Noetic on Raspberry Pi
Ubuntu 20.04 Terminal
$sudo sh -c 'echo "deb http://packages.ros.org/ros/ubuntu focal main" > /etc/apt/sources.list.d/ros-latest.list'
Adds the official ROS package repository to the system's software sources list. Without this, apt does not know where to find ROS packages because they are not in the standard Ubuntu repositories. focal is the codename for Ubuntu 20.04.
$sudo apt install ros-noetic-desktop-full -y
Installs the full ROS Noetic desktop environment. This takes around 15 minutes on the Pi. Run this once during the initial hardware setup, not before every flight test.
$echo "source /opt/ros/noetic/setup.bash" >> ~/.bashrc
$source ~/.bashrc
The setup.bash script adds all ROS commands (roscore, rosrun, roslaunch, etc.) to the system PATH and sets all required environment variables. Adding it to .bashrc means it runs automatically every time a terminal opens, so the ROS commands are always available without needing to source it manually.
Step 2 — Install MAVROS
Terminal
$sudo apt install ros-noetic-mavros ros-noetic-mavros-extras -y
Installs MAVROS and its extras in one command.
$wget https://raw.githubusercontent.com/mavlink/mavros/master/mavros/scripts/install_geographiclib_datasets.sh
$sudo bash install_geographiclib_datasets.sh
Downloads and installs the GeographicLib datasets that MAVROS requires for Earth coordinate frame conversions. This step is frequently missed by beginners and results in MAVROS refusing to start. It only needs to be run once per machine.
Step 3 — Build the ROS Workspace
Terminal (run from the ros_ws directory)
$cd ~/quadcopter_smc_project/05_hardware/ros_ws/
$catkin_make
catkin_make is the ROS build command. It reads CMakeLists.txt inside our smc_controller_pkg, compiles any C++ code (there is none here since we use Python, but the manifest still needs building), and sets up the workspace so that roslaunch can find our package. This must be re-run every time a new Python file is added to the package or the CMakeLists.txt is changed.
$source devel/setup.bash
Activates the workspace so that ROS can find our custom package. Unlike the system setup.bash (added to .bashrc), this workspace setup.bash must be sourced every time a new terminal is opened for development. Add it to .bashrc once the workspace is stable and no longer changing frequently.
Pre-Flight Check Commands (run before every flight test)
Terminal
$rostopic echo /mavros/state
Prints the current Pixhawk connection state in real time. The connected field must show true, and the armed field should show false before any test begins. If connected is false, the USB/UART cable is not properly connected or MAVROS is not receiving heartbeat packets from PX4.
$rostopic echo /mavros/imu/data | head -20
Prints the first 20 lines of attitude data coming from the Pixhawk IMU. Use this to confirm that the attitude estimates look reasonable (all angles near zero when the drone is sitting level on a table) and that the update rate looks healthy (timestamps advancing at roughly 4ms intervals for 250 Hz).
$rostopic hz /mavros/imu/data
Measures the actual publishing rate of the IMU topic in Hz. The reported rate should be close to 250 Hz. If it is significantly lower (e.g. 50 Hz), the MAVROS configuration needs adjusting — the IMU stream rate can be set in the MAVROS launch file using the imu_raw_pub_rate parameter.
$roslaunch smc_controller_pkg smc_flight.launch
The single command that starts everything for a flight test. The launch file starts MAVROS with the correct connection parameters, then starts our smc_node.py with the config.yaml parameters loaded. All output from both nodes appears in the same terminal, colour-coded by node name. This is the command used at the start of every flight test in Phase 6.
Debugging Commands (used when something goes wrong)
Terminal
$rosnode list
Lists all currently running ROS nodes. After launching, you should see /mavros and /smc_controller in the list. If one is missing it means that node crashed at startup, and you need to check its output for error messages.

Opens a live scrolling plot of the roll rate from the IMU. Extremely useful during bench testing (propellers removed) to visually confirm that the controller is responding correctly to manually tilting the drone before any motors are armed.

Section 09

PX4 Autopilot and QGroundControl

PX4 is the open-source flight controller firmware running on the Pixhawk 4. QGroundControl (QGC) is the ground station application used to configure and calibrate PX4. Together they form the hardware interface layer that our software commands.

Flight Controller
PX4 Autopilot v1.14
Open-source flight controller firmware
PX4 is the firmware that runs directly on the Pixhawk microcontroller. It handles all hardware-level operations: reading the IMU at 1000 Hz and filtering it, running the Extended Kalman Filter for state estimation at 250 Hz, generating ESC PWM signals at 400 Hz, monitoring battery voltage and GPS signal, and enforcing all safety failsafes. For this project we run PX4 in OFFBOARD mode, which means it accepts attitude and actuator control commands from our Raspberry Pi SMC node over MAVLink and executes them as given, while still monitoring for signal loss and battery failsafe conditions.
OFFBOARD mode EKF state estimation MAVLink server ESC PWM Safety failsafe ULog recording
Ground Station
QGroundControl
GUI application for PX4 configuration and monitoring
QGroundControl is the desktop application used on a laptop to configure the Pixhawk over USB. Every calibration, every parameter change, every firmware flash goes through QGC. It provides a graphical interface for the six mandatory calibration steps (accelerometer, gyroscope, magnetometer, level horizon, radio transmitter, ESC), a parameter editor for searching and setting the thousands of PX4 configuration parameters, a real-time attitude indicator for monitoring the EKF output, and log download for retrieving ULog files after flight. It runs on Windows, macOS, and Ubuntu.
Firmware flash Sensor calibration Parameter editor Log download Live telemetry

Critical PX4 Parameters to Configure

COM_OF_LOSS_T — set this to 1.0 second. This is the OFFBOARD connection timeout. If the Raspberry Pi stops sending MAVLink heartbeats for more than this duration (because our node crashed), PX4 automatically switches back to manual mode rather than continuing to execute the last received command indefinitely. This is the primary software safety net.

SDLOG_PROFILE — set this to include the Full log profile. This ensures that all attitude, actuator, and sensor topics are recorded in the ULog at maximum rate. The default logging profile omits some topics that are needed for the Phase 6 analysis.

IMU_GYRO_RATEMAX — set this to 400 Hz. This controls how fast PX4 publishes gyroscope data via MAVLink. A higher rate ensures our 250 Hz SMC loop always has fresh measurements and does not use stale data from the previous time step.

COM_RCL_EXCEPT — set this to 4 (OFFBOARD mode). This tells PX4 that it is normal to not have a radio transmitter signal when in OFFBOARD mode and prevents it from triggering a failsafe the moment our RC transmitter is switched off during an autonomous test.

Configure Serial Port for MAVROS on Raspberry Pi
Terminal
$ls /dev/ttyUSB* /dev/ttyACM*
Lists all USB serial devices. When the Pixhawk is connected via USB, it typically appears as /dev/ttyACM0. This device name goes into the MAVROS launch file as the connection URL.
$sudo usermod -a -G dialout $USER
Adds the current user to the dialout group, which has permission to access serial port devices. Without this, MAVROS will fail to open /dev/ttyACM0 with a Permission denied error. A logout and login is required after this command for the group change to take effect.
$sudo chmod 666 /dev/ttyACM0
Grants read and write permission to all users for the Pixhawk serial port. This is a temporary fix if the usermod approach has not yet taken effect. The permission resets when the cable is unplugged and re-plugged, so the usermod approach is the proper long-term solution.

Section 11

How All the Languages Connect Together

The four languages and the hardware form a data pipeline. Understanding this pipeline is essential because any error at one stage affects everything downstream. Here is the complete end-to-end flow from sensor to motor, and from raw data to published result.

Real-time control loop (runs at 250 Hz during flight)
Pixhawk Hardware

IMU reads angular velocity and acceleration at 1000 Hz

The MPU-6000 IMU sensor inside the Pixhawk 4 samples the gyroscope and accelerometer at 1000 Hz. PX4 firmware fuses these samples with magnetometer and barometer readings in its Extended Kalman Filter, running at 250 Hz, to produce a smooth estimate of roll, pitch, yaw, and their rates. This fused estimate is what gets sent to our SMC node, not the raw sensor values.

PX4 C++ firmware Hardware layer
↓   MAVLink serial (USB)   ↓
MAVROS Node

MAVROS converts MAVLink packets to ROS topics

The MAVROS ROS node runs continuously on the Raspberry Pi, reading the incoming MAVLink serial stream from the Pixhawk. Every attitude estimate packet is converted into a sensor_msgs/Imu ROS message and published to the /mavros/imu/data topic. Our SMC node subscribes to this topic and gets called automatically every time a new message arrives, which is 250 times per second.

ROS / MAVROS C++ (MAVROS internal)
↓   ROS topic publish/subscribe   ↓
SMC Node (Python)

Python runs the SMC algorithm on the received attitude data

The smc_node.py callback function fires each time new IMU data arrives. It extracts the roll, pitch, yaw angles and angular rates from the ROS message, passes them to smc_algorithm.py, which computes the sliding surface values using NumPy, evaluates the equivalent control from the stored nominal dynamics, adds the saturation-function switching control, and returns three torque commands. The entire computation takes under 0.5 milliseconds on the Raspberry Pi 4, well within the 4-millisecond budget.

Python 3.10 NumPy ROS
↓   motor_mixer.py converts torques to motor speeds   ↓
MAVROS Publish

Motor speed commands published back to Pixhawk over MAVLink

The four individual motor speed commands (normalised between 0 and 1) are packed into an ActuatorControl ROS message and published to the /mavros/actuator_control topic. MAVROS converts this message back into a MAVLink ACTUATOR_CONTROL_TARGET packet and sends it over the serial link to the Pixhawk. PX4 maps the normalised values to ESC PWM signals (typically 1000 to 2000 microseconds) and sends them to each ESC at 400 Hz.

ROS / MAVROS PX4 firmware
↓   PWM signals   ↓
ESCs and Motors

ESCs drive the brushless motors at the commanded speeds

Each of the four 30A electronic speed controllers receives a PWM signal from the Pixhawk and drives its brushless motor at the corresponding speed. The spinning propellers generate the thrust forces and reactive torques that physically rotate the drone. The IMU then measures this resulting rotation and the loop completes back at the top. The entire loop from measurement to actuation takes under 4 milliseconds, giving a feedback bandwidth of 250 Hz.

Offline analysis pipeline (runs after each flight test)
Post-flight

Python converts ULog to CSV → MATLAB analyses and plots → Report figures

After each flight test, the ULog binary file is downloaded from the Pixhawk over USB using QGroundControl. convert_logs.py (Python + pyulog) batch-converts all ULog files to CSV. analyse_flight.m (MATLAB) loads the CSV files, computes all performance metrics, runs the paired t-test, and saves a results_summary.csv. generate_figures.m then produces all publication-quality PDF plots for the Phase 7 report. This three-step pipeline (Python for conversion, MATLAB for analysis, MATLAB for plotting) is clean and reproducible.

Section 12

Which Tool Is Used in Which Phase

A quick reference showing exactly which part of the repository and which tools are active in each of the seven project phases.

Phase 1
Folder 01_model. Tools: MATLAB only. Toolboxes: Symbolic Math (for verifying coupling terms), ODE45 (for simulation). No hardware, no Python, no ROS. The only output is a set of validated model plots confirming the ODE is physically correct.
Phase 2
Folder 02_pid_baseline. Tools: MATLAB and Simulink. Toolboxes: Control System Toolbox, Simulink Control Design Toolbox. The pid_simulink.slx Simulink model is built here. Outputs are PID gains, step response plots, and the first disturbance rejection result to use as the baseline benchmark.
Phase 3
Folder 03_smc_design. Tools: MATLAB only. Toolboxes: Symbolic Math Toolbox for Lyapunov verification. No Simulink yet. The output of this phase is the complete analytical SMC control law and a verified stability proof. Everything is still mathematical at this stage.
Phase 4
Folder 04_simulation. Tools: MATLAB and Simulink. Toolboxes: Statistics Toolbox for RMSE. The full_model.slx Simulink file is the central deliverable. All five test cases are run and both PID and SMC results are compared. This phase produces the simulation half of the final results chapter.
Phase 5
Folder 05_hardware. Tools: QGroundControl (for PX4), Python and ROS on Raspberry Pi, MATLAB for thrust stand data fitting. This is the most complex phase involving all four tools simultaneously. Hardware is assembled, PX4 is configured, the ROS workspace is built, and the first bench tests happen with propellers removed.
Phase 6
Folder 06_experimental. Tools: All hardware tools for flight tests, Python for log conversion, MATLAB for analysis. The smc_flight.launch command is the start of every test session. After testing, convert_logs.py runs first, then analyse_flight.m in MATLAB. The experimental results chapter is built entirely from this folder.
Phase 7
Folder 07_report. Tools: MATLAB for generating final figures from all previous phases, Microsoft Word for writing, BibTeX for references. No new experiments, no code changes. All figures are regenerated from existing data using generate_figures.m scripts to ensure they are all consistent in style and format.
